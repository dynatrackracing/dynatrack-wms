## Shell
- **SPA**: a fixed nav + left sidebar + `<main>` holding **eleven** `.page` divs (Labels removed; Pick List added 2026-05-28; Shipped Items added 2026-05-29; **Pick List · Errors added 2026-05-30**). **All eleven `.page` divs are direct children of `<main>` (siblings)** — verified by a depth probe (div balance 319/319). ⚠️ *2026-05-29 fix:* `#page-admin` was missing its closing `</div>`, so `#page-health` was swallowed as a child of `#page-admin` and stayed hidden (Inventory Health blank page) whenever Admin wasn't active; the missing `</div>` was restored. `navigate(p)` (693) toggles `.active` on `#page-<p>` and the matching sidebar `.ni[data-page]`. An **`@media print`** block (in `<style>`) hides `nav`/`aside`/other pages/`.no-print` so only the active page prints (used by Pick List).
- **Auth**: login screen (`#login-screen`, ~140). Token stored in `localStorage` as `wms_token`; sent as `x-wms-token` by the `api()` helper (635). A 401 from `api()` triggers `showLogin()` + throws "Session expired…".
- **Top status dot** (nav) reflects **DB health only** (honest, 2026-05-28). `setSyncing()`/`setSynced()` (653/657) show transient "Syncing…/Live". **`refreshDbStatus()` (663)** drives the authoritative state from `/api/health`'s `db` field → Live / **DB Error** (only when `db≠connected`) / Offline; it's called on init + after sign-in. **`setSyncErr(err)` (677)** classifies: a 401/"session expired" → amber **"Session expired"** (`.sync-dot.warn`); any other failure → defers to `refreshDbStatus()` (so a failed eBay sync with a healthy DB stays "Live" — surfaced via toast + per-store cards, NOT the DB dot). All catch sites pass the error. Toasts via `showToast(msg,isErr)` (710).

## Key data facts (gotchas for future sessions)
- **eBay orders & listings are NOT persisted** — they live only in the in-memory JS arrays `ORDERS` and `ALL_LISTINGS`, repopulated per **live sync** and **reset on page refresh**. Inventory Health depends on `ALL_LISTINGS` being loaded first (hence the empty-state Sync button).
- **CSV import retired 2026-05-28** — the Dashboard "Quick Import" card and the Import-CSV buttons/file-inputs on eBay Orders & Listings were removed, along with `parseCSV`/`importOrdersCSV`/`importListingsCSV`. CSV import was fully client-side (FileReader) — no backend upload route existed. **Export CSV is retained** (Admin `exportCSV` + Inventory Health `exportHealthCSV`).
- **Multi-store:** every `ORDERS`/`ALL_LISTINGS` entry carries a `store` tag (`dynatrack`/`autolumen`). `syncEbayOrders`/`syncEbayListings` hit the combined endpoints (full replace — both stores). **`syncStore(key)` (per-store) MUST replace only that store's slice via `arr.filter(x => x.store !== key).concat(fresh)` — never reassign the array, or the other store is silently wiped** (commented at the call site). `storeBadge`/`storeLabel`/`storeCountLabel` render store chips/counts.
- **SKU matching (Rule 8) — now TOP-LEVEL, with a multi-serial tokenizer (2026-05-31):** `normalizeSkuKey(sku)` (hoisted out of `loadInventoryHealth` to a top-level fn) `(sku||'').trim().toUpperCase().replace(/[A-Z*]+$/,'')` strips a trailing run of letters AND/OR asterisks (`INT4306R`→`INT4306`; `INT4306V*`/`INT4306*`/`INT3980V**`→`INT3980`/`INT4306`; `*` added 2026-06-04 so `*`-suffixed serials match — digit-terminated/hyphenated junk untouched, so `isIncompleteKey` still flags real junk). **`listedSerialKeys(field)`** resolves an eBay "Custom label (SKU)" field that packs MULTIPLE one-of-one serials into one string → the array of WMS serial keys it lists: splits on whitespace, **prefix-inheritance** (a bare numeric token inherits the last alpha prefix — `"MOD17043V 17075V 17587"`→`[MOD17043,MOD17075,MOD17587]`; verified 78/91 bare tokens match inherited vs 9 as-is), strips store tags (`AUTOLUMEN`/`DYNATRACK`/typo `AUTOLUMENA`), internal-id suffix (`MOD10131/000046`→`MOD10131`), trailing `*`/commas, qty markers (`(3)`, `MOD16197(3)`), and skips pure-junk words (`Garage`/`Ford`/`HOLD`/`?`). `loadInventoryHealth`'s `ebayByKey` now registers each live listing under EVERY key `listedSerialKeys` returns, so a multi-serial listing is "listed" for all its component parts (rescued 289 STORED items from false-unlisted; eBay-Only 348→175). ⚠️ **`normalizeSkuKey` MUST stay byte-identical to the server.js copy** (used by `reconcileOrderLines`); true centralization is blocked by the no-build single-file frontend (Rule 18, #14). **`listedSerialKeys` is frontend-only** (the matcher is client-side); the order-reconcile still matches single `line.sku` — when the PARKED multi-serial pick/ship work lands (#21 scan-verify), mirror `listedSerialKeys` into server.js byte-identical.
- **Pick List (age-aware rework 2026-05-30):** `loadPickList` now renders only `GET /api/picklist`'s **`active`** group (≤3 business days); stale (>3 bd) lines route to the new **`#page-picklist-errors`** page (`loadPickListErrors`), which also shows the **Dismissed** archive (`GET /api/picklist/dismissed`). **Dismiss** (`dismissPickLine`→`POST /api/picklist/dismiss`) moves a stale line into the retained DISMISSED archive (out of the daily flow, never deleted); **Restore** (`restorePickLine`→`POST /api/picklist/restore`) puts it back. `updateErrorsNavBadge(n)` drives the dimmed sidebar "Errors" entry's red count badge. Print on `#page-picklist` still prints just the active 3-col sheet. (Prior S4 state: VIEW+PRINT only, `markPicked`/`POST /api/pick` already removed; items ship via `reconcileOrderLines` Phase 2.)
- All data access goes through `api(path,opts)` (635) → `fetch('/api'+path)` with the token header. No third-party SDKs (rule 19).
- **Top status dot = DB health only** (`refreshDbStatus`, 663). It is NOT a generic op spinner: eBay/auth/network failures must not turn it red "DB Error". A 401 → amber "Session expired"; non-DB op failures defer to `/api/health.db`. Only `db≠connected` shows "DB Error". (Fixed 2026-05-28 — `setSyncErr` used to hardcode "DB Error" for any caught failure.)
- **Labels page removed 2026-05-28** (serials/barcodes are minted in a separate external system, scanned in via Scan). With it gone, two backend routes have **no frontend caller**: `POST /api/sequences/next/:prefix` and `GET`+`POST /api/print-log` (orphaned — left in server.js pending a separate decision). `GET/POST/PATCH /api/sequences` are still used by Admin's Serial Sequences section. The `.lp*` label-preview CSS classes are now dead (harmless, left in place).
