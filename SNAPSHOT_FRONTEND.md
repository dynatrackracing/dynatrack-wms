## Shell
- **SPA**: a fixed nav + left sidebar + `<main>` holding **ten** `.page` divs (Labels removed; Pick List added 2026-05-28; **Shipped Items added 2026-05-29**). **All ten `.page` divs are direct children of `<main>` (siblings)** вЂ” verified by a depth probe. вљ пёЏ *2026-05-29 fix:* `#page-admin` was missing its closing `</div>`, so `#page-health` was swallowed as a child of `#page-admin` and stayed hidden (Inventory Health blank page) whenever Admin wasn't active; the missing `</div>` was restored. `navigate(p)` (693) toggles `.active` on `#page-<p>` and the matching sidebar `.ni[data-page]`. An **`@media print`** block (in `<style>`) hides `nav`/`aside`/other pages/`.no-print` so only the active page prints (used by Pick List).
- **Auth**: login screen (`#login-screen`, ~140). Token stored in `localStorage` as `wms_token`; sent as `x-wms-token` by the `api()` helper (635). A 401 from `api()` triggers `showLogin()` + throws "Session expiredвЂ¦".
- **Top status dot** (nav) reflects **DB health only** (honest, 2026-05-28). `setSyncing()`/`setSynced()` (653/657) show transient "SyncingвЂ¦/Live". **`refreshDbStatus()` (663)** drives the authoritative state from `/api/health`'s `db` field в†’ Live / **DB Error** (only when `dbв‰ connected`) / Offline; it's called on init + after sign-in. **`setSyncErr(err)` (677)** classifies: a 401/"session expired" в†’ amber **"Session expired"** (`.sync-dot.warn`); any other failure в†’ defers to `refreshDbStatus()` (so a failed eBay sync with a healthy DB stays "Live" вЂ” surfaced via toast + per-store cards, NOT the DB dot). All catch sites pass the error. Toasts via `showToast(msg,isErr)` (710).

## Key data facts (gotchas for future sessions)
- **eBay orders & listings are NOT persisted** вЂ” they live only in the in-memory JS arrays `ORDERS` and `ALL_LISTINGS`, repopulated per **live sync** and **reset on page refresh**. Inventory Health depends on `ALL_LISTINGS` being loaded first (hence the empty-state Sync button).
- **CSV import retired 2026-05-28** вЂ” the Dashboard "Quick Import" card and the Import-CSV buttons/file-inputs on eBay Orders & Listings were removed, along with `parseCSV`/`importOrdersCSV`/`importListingsCSV`. CSV import was fully client-side (FileReader) вЂ” no backend upload route existed. **Export CSV is retained** (Admin `exportCSV` + Inventory Health `exportHealthCSV`).
- **Multi-store:** every `ORDERS`/`ALL_LISTINGS` entry carries a `store` tag (`dynatrack`/`autolumen`). `syncEbayOrders`/`syncEbayListings` hit the combined endpoints (full replace вЂ” both stores). **`syncStore(key)` (per-store) MUST replace only that store's slice via `arr.filter(x => x.store !== key).concat(fresh)` вЂ” never reassign the array, or the other store is silently wiped** (commented at the call site). `storeBadge`/`storeLabel`/`storeCountLabel` render store chips/counts.
- **SKU normalization** (`normalizeSkuKey`, frontend 1342): `(sku||'').trim().toUpperCase().replace(/[A-Z]+$/,'')` вЂ” strips trailing letters so eBay `INT4306R` matches WMS serial `INT4306` (rule 8). вљ пёЏ **A byte-identical copy now also lives server-side** (`normalizeSkuKey` in server.js, used by `/api/picklist`) вЂ” keep the two in sync until centralized.
- **Pick List (rebuilt 2026-05-29 S4 вЂ” VIEW+PRINT only):** `loadPickList`/`pickRow`/`printPickList` drive `#page-picklist` off `GET /api/picklist`, which is now a **flat read of `ebay_order_lines` (NEEDS_PICK)** вЂ” no live order join, no client mutation. Renders ONE 3-column table (LOCATION В· SKU В· ITEM DESCRIPTION), location-unknown items grouped last. **`markPicked` and `POST /api/pick` were removed** вЂ” items are shipped automatically by the eBay orders sync (`reconcileOrderLines` Phase 2), not from this page.
- All data access goes through `api(path,opts)` (635) в†’ `fetch('/api'+path)` with the token header. No third-party SDKs (rule 19).
- **Top status dot = DB health only** (`refreshDbStatus`, 663). It is NOT a generic op spinner: eBay/auth/network failures must not turn it red "DB Error". A 401 в†’ amber "Session expired"; non-DB op failures defer to `/api/health.db`. Only `dbв‰ connected` shows "DB Error". (Fixed 2026-05-28 вЂ” `setSyncErr` used to hardcode "DB Error" for any caught failure.)
- **Labels page removed 2026-05-28** (serials/barcodes are minted in a separate external system, scanned in via Scan). With it gone, two backend routes have **no frontend caller**: `POST /api/sequences/next/:prefix` and `GET`+`POST /api/print-log` (orphaned вЂ” left in server.js pending a separate decision). `GET/POST/PATCH /api/sequences` are still used by Admin's Serial Sequences section. The `.lp*` label-preview CSS classes are now dead (harmless, left in place).
