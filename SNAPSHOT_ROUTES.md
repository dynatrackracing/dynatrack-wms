# SNAPSHOT_ROUTES.md — `server.js` API surface

> Orientation map of the backend. Regenerate at session end if `server.js` changed (HAWKER_RULES rule 38).
> Generated 2026-05-29 from `server.js` (single Express file, ~878 lines). Line numbers are approximate anchors.

## Architecture
- Single-file Express app (`server.js`) + PostgreSQL (`pg.Pool`, `DATABASE_URL`). Serves the SPA from `public/` and all `/api/*` routes.
- **Auth:** token-based. `POST /api/login` checks `WMS_USERNAME`/`WMS_PASSWORD` (env) and returns a random 64-char hex token. **Persisted in the `sessions` table (migration 0006, 2026-06-03) — was an in-memory Map that dropped every login on restart; now survives deploys/restarts.** 12h **sliding** TTL: `requireAuth` (and `/api/me`) call **`touchSession`** = one atomic read-and-slide `UPDATE sessions SET expires_at=NOW()+12h WHERE token=$1 AND expires_at>NOW() RETURNING username` (row → authed + window advanced; none → 401). Reads the **`x-wms-token`** header. Hourly sweep `DELETE WHERE expires_at<=NOW()`. No new env var; token stored raw. *(All auth handlers + `requireAuth` are now `async`.)*
- **Env vars:** `PORT`, `DATABASE_URL`, `WMS_USERNAME` (default `admin`), `WMS_PASSWORD` (required — login blocked if unset). **eBay is multi-store:** per-store creds via prefixed vars `DYNATRACK_TRADING_API_{APP_NAME,CERT_NAME,DEV_NAME,TOKEN}` and `AUTOLUMEN_TRADING_API_*`. The legacy un-prefixed `TRADING_API_*` set is **IGNORED** (no fallback) — kept only as a single-store rollback safety net.

## eBay helper layer (322–535) — multi-store
- **`STORES` registry (331):** `[{key,label,prefix}]` for `dynatrack`/`autolumen`. **Adding a 3rd store = one more entry.** `getStore`, `storeCreds(key)` (reads `${prefix}_TRADING_API_*` — **no un-prefixed fallback**), `missingStoreVars`/`storeConfigured`.
- **`validateStoreEnv()` (360)** runs at boot: loud per-store `OK`/`[MISCONFIG]` log + an explicit "legacy un-prefixed TRADING_API_* detected but IGNORED" line. **Soft disable, never throws** — a fat-fingered eBay cred can't crash warehouse ops; a misconfigured store's routes fail loud on call instead.
- **`ebayHeaders(store, callName)` (376)** / **`ebayCall(store, callName, bodyXml)` (404):** `store` is **required, no default, no shared cred path** — calls can't silently use the wrong store. `ebayCall` throws `store '<k>' not configured: missing …` when creds absent.
- **Per-store fetch helpers:** `fetchStoreHealth` (439, never throws — returns tagged status), `fetchStoreListings` (463), `fetchStoreOrders` (504). Combined routes fan out over `STORES` via these; per-store routes call one.
  - **`fetchStoreOrders` (504)** now requests `DetailLevel=ReturnAll` and parses order-level fields from the **head** (everything before `<TransactionArray>`, so a transaction's own `<Status>`/`<ShippedTime>` can't be mistaken for the order's). Each order keeps its existing shape (`id,status,shipped,buyer,total,date,items[]`) **plus additive fields** — order-level `shippedTime,paidTime,checkoutStatus,paymentStatus,lastModified`; each `items[]` line gains `orderLineItemId,itemId,transactionId,lineShippedTime`. `/api/picklist` ignores the additions (shape preserved).
- `parseXmlValue` / `parseXmlAll` are regex-based minimal XML extractors.
- ⚠️ **`ebayCall` ignores `res.statusCode`** — non-200/HTML body resolves as data; `/api/ebay/health` guards via the no-`<Ack>` check, listings/orders surface it as a thrown "eBay API error".
- ✅ **Cross-contamination verified 2026-05-28:** dynatrack (3,272) vs autolumen (532) listings, disjoint ItemIDs, overlap 0 — per-store creds isolated.

## Pick List layer (643–835)
- **`normalizeSkuKey(s)` (854):** `(s||'').trim().toUpperCase().replace(/[A-Z*]+$/,'')` — server-side Rule-8 normalization; strips a trailing run of letters AND/OR asterisks (`INT4306R`/`INT4306V*`/`INT4306*`/`INT3980V**`→bare; `*` added 2026-06-04 so `*`-suffixed sold SKUs normalize/match — digit-terminated & hyphenated/numeric junk untouched). ⚠️ **MUST stay byte-identical** (the `.replace` regex) to the frontend copy in public/index.html. (Centralize later.) Used by `reconcileOrderLines` (matching). *(No longer used by `/api/picklist`, which now reads the persisted `matched_serial`.)*
- `GET /api/picklist` (979) reads `ebay_order_lines` (NEEDS_PICK) and **splits by business-day age** into `active`/`errors` — no live eBay call, no mutation. See the route table.
- **Business-day aging (added 2026-05-30):** `HOLIDAYS` (929) = an editable `Set` of `'YYYY-MM-DD'` US federal-holiday dates (seeded 2026–2027). `businessDaysSince(from, now)` (960) counts US-Eastern weekdays (Mon–Fri) minus HOLIDAYS in the half-open interval `(fromDay, today]` — both ends reduced to their `America/New_York` calendar date, day-stepped from a noon-UTC anchor (DST-safe). Paid today → 0. ⚠️ relies on Node full-ICU (`timeZone` in `toLocaleDateString`) — present in Node 18 default build.
- **Dismiss/restore (added 2026-05-30):** `POST /api/picklist/dismiss` + `/restore` are the **only writers of `disposition` outside the reconcile** — both single guarded `UPDATE`s on `ebay_order_lines` (no `items`/`moves`, no eBay). DISMISSED is a manual decision the reconcile never overwrites (ON CONFLICT `WHEN …='DISMISSED' THEN 'DISMISSED'`), so a dismissed line — even if eBay later ships it — stays DISMISSED until manually restored (it is NOT auto-ship-moved while dismissed).
- **`POST /api/pick` was REMOVED (2026-05-29 S4).** The pick→ship action no longer exists as a route; shipping is done automatically by `reconcileOrderLines` Phase 2 when eBay reports a line shipped. The Pick List is **view+print only**. (Its sole caller, the frontend `markPicked`, was removed in the same session.)
- **`moves.to_location='SHIPPED'` + `items.location='SHIPPED'`** now reference the **real** `'SHIPPED'` location row (id 2713). The ship-move (reconcile Phase 2) writes both. `moves.to_location` still has no FK, but the value is now a real location.

## Order-line reconcile (`reconcileOrderLines`, 668) — two phases (added 2026-05-29)
Runs as a **side-effect of the two order-sync routes** (`/api/ebay/orders`, `/api/ebay/:store/orders`). Pushes nothing to eBay (Rule 25). Returns `{upserts, skipped, moved}`.
- **PHASE 1 — populate.** Upserts fetched eBay order **lines** into **`ebay_order_lines`** keyed by `order_line_item_id` (eBay `OrderLineItemID`; falls back to `ItemID-TransactionID`, skips a line that has neither). Upsert-only (never deletes).
  - **Derived state:** `paid` = OrderStatus `Completed` ∧ CheckoutStatus.Status `Complete` ∧ eBayPaymentStatus `NoPaymentFailure` ∧ PaidTime present; `shipped` = order-level ShippedTime ∨ that line's `Transaction.ShippedTime`; `cancelled` = OrderStatus `Cancelled`/`CancelPending` ∨ **`CancelStatus`=`CancelComplete`** ∨ **a SUCCEEDED refund (`MonetaryDetails`→`RefundStatus`=`Succeeded`)** ∨ (checkout flipped to `Incomplete` while paid). **(refund-detection fix 2026-05-30:** eBay leaves a refunded order `OrderStatus=Completed`/`CheckoutStatus=Complete`, so the refund node is the ONLY cancel signal — without it, refunded-but-unshipped lines were stuck in NEEDS_PICK. `fetchStoreOrders` now parses `cancelStatus`/`refundStatus`. Ship wins over cancel, so a *shipped-then-refunded* return stays SHIPPED — refund only cancels an **un**shipped line, whose item correctly stays STORED.)
  - **`disposition`:** shipped→`SHIPPED`; else cancelled→`CANCELLED`; else paid→`NEEDS_PICK`; else the line is skipped (unpaid/open checkout — not actionable yet).
  - **Matching:** `sku_norm` → **active STORED** `items.serial` (`status='STORED' AND archived_at IS NULL` — archived/scrapped items are **not** pick candidates, 2026-05-30). Exactly 1 → `matched_serial`; 0 or >1 → `location_unknown=true` + `matched_serial` NULL (>1 is ambiguous — do **not** guess). Lines are never dropped. (Phase-2 ship-move likewise skips archived items.)
  - **MONOTONIC upsert (ON CONFLICT):** a row already `SHIPPED`/`CANCELLED`/`DISMISSED` is never pulled back to `NEEDS_PICK`; `DISMISSED` (a manual decision) is never overwritten by the sync. `shipped`/`paid` are sticky-true (`OR`); `ebay_shipped_time`/`paid_time`/`title`/`matched_serial`/`ebay_last_modified` use `COALESCE` (a later null never wipes a known value); `last_synced=NOW()` each pass. Chunked (500 rows) inside one txn.
  - **SHIP-ONCE SELF-HEAL (2026-07-07):** the same ON-CONFLICT upsert also sets **`ship_move_applied_at`** when this (effectively) `SHIPPED` line's matched item is **already `SHIPPED`** — `COALESCE(ebay_order_lines.ship_move_applied_at, CASE WHEN (EXCLUDED.disposition='SHIPPED' OR ebay_order_lines.disposition='SHIPPED') AND ebay_order_lines.disposition IS DISTINCT FROM 'DISMISSED' AND EXISTS(SELECT 1 FROM items it WHERE it.serial=COALESCE(EXCLUDED.matched_serial, ebay_order_lines.matched_serial) AND it.status='SHIPPED') THEN NOW() END)`. It only ever sets NULL→`NOW()` (never un-stamps), evaluates in SQL because a re-synced already-shipped item is no longer a STORED match candidate (so its `matched_serial` lives only in the DB row), and does **not** touch disposition / ship-wins-over-cancel. **Why:** an item shipped by **other means — e.g. a manual move to `SHIPPED` (`moved_by='dynatrack'`), which never runs Phase 2's stamp — used to leave its SHIPPED line `ship_move_applied_at IS NULL`, so on a later return-to-STORED Phase 2 re-shipped it once (the pre-0004 symptom, re-armed). This self-heals every such line on the next sync and stops new landmines forming.
- **PHASE 2 — ship-move (the only place the reconcile mutates `items`/`moves`). SHIP-ONCE PER LINE (migration 0004, 2026-05-31).** Candidates = items still `STORED` (+ `archived_at IS NULL`) with ≥1 matched `SHIPPED` line **whose `ship_move_applied_at IS NULL`** (one row per item). Each runs **ONE audited txn mirroring `/api/move`**: re-check `status='STORED'` `FOR UPDATE`, ensure the `'SHIPPED'` location row, `UPDATE items SET status='SHIPPED', location='SHIPPED'`, INSERT exactly one `moves` row (`from_location`=prior shelf → `'SHIPPED'`, `moved_by='ebay-sync'`), **and in the SAME txn `UPDATE ebay_order_lines SET ship_move_applied_at=NOW()` for that serial's unstamped SHIPPED line(s)** (atomic — rolls back together). **Why:** before this, Phase 2 re-shipped ANY STORED item with a matched SHIPPED line, so a **returned item scanned back to STORED was re-shipped on every sync** (its order line stays SHIPPED in eBay's 90-day window). Now the line ships its item once; a later return to STORED is left alone; a genuine re-sale is a new OLI (fresh NULL line) → ships once. Phase 1's ON CONFLICT now **also self-heals** `ship_move_applied_at` for already-SHIPPED matched items (2026-07-07, see Phase 1 above) — but still never *un-stamps* it. **Idempotent + monotonic** as before (STORED `FOR UPDATE` guard; `location_unknown`/ambiguous never move). Two one-time backfills: the original 0004 deploy stamped all 1,842 then-SHIPPED lines; a second (2026-07-07) stamped the **608** unstamped SHIPPED lines that had accrued since (chiefly from manual ship-moves that never ran the Phase-2 stamp) → 0 unstamped SHIPPED lines remain. **This is the only ship path** (no `/api/pick`, removed S4).
